XamarinApp

Ce dossier contient 3 sous-dossiers :

  • XamApp : contient tout ce qui est partagé par le développement pour Android et iOS. Ceci contient je dirais au moins 75 % du code de l’app.
  • XamApp.Android : contient tout ce qui est spécifique à Androïde.
  • XamApp.iOS : contient tout ce qui est spécifique à iOS. Le développement requit d’avoir un MacBook, ou plus précisément un ordi avec Xcode démarré sur ça.

Ceux qui veulent participer au développement de l'application mobile, veuillez d'abord suivre ce petit cours.

WebAPI

Ce dossier contient le code source du serveur. Le déploiement du serveur mettra la version compiler sur Azure Portal. Là-bas les Azure Resources suivantes seront utilisées :

  • App Service : pour loger le déploiement du WebAPI
  • SQL Server / Serveur SQL : pour loger la base des données
  • Blob Storage / Compte de Stockage : pour loger les fichiers (envoyés par les utilisateurs de l’app)
  • Notification Hub : pour faire le transfert des Push Notifications (que l’app utilise pour informer l’utilisateur qu’un message vient d’arriver)

Le développement en local, c'est-a-dire sur son ordinateur sans connexion Internet, utilise un serveur SQL local et un émulateur de compte de stockage local. Il n’y a pas de local pour le Notification Hub, mais l'on peut se connecter sur Internet juste pour utiliser le Notification Hub sur Azure.

Ceux qui veulent participer au développement du serveur, veuillez d'abord suivre ce long cours.

Global

Ce dossier contient un simple Visual Studio Library-Project qui sert à synchroniser l’app avec le serveur.

Par exemple le sous-dossier Models contient les Data Transfer Objects (DTOs) dont leurs données sont converties en textes (c’est-a-dire JSON serialized) par le serveur puis envoyées à l’app comme réponse à une requête HTTP fait par l’app. L’app va recevoir le texte et le convertir au même DTO (c’est-a-dire JSON deserialize) puis utiliser les données obtenues.

L’Utilisation de Git / GitHub

Pour tout travail important où l'on saisie du texte, il est toujours important d'utiliser un logiciel de gestion de versions. Git est présentement le plus populaire. Vous devez connaître à propos de Git pour comprendre la suite!

La branche principale

Sur Git il y a bien sûr la branche master qui est la branche principale du répertoire. Cette branche est renommée main car le nom main est beaucoup plus descriptive. Je ne suis pas du tout d'accord avec la raison de GitHub pour ce changement de nom. Toutefois, je trouve que main est plus descriptif que master.

La branche principale contient la version à-jour de tous les travaux développés dans le répertoire du projet. Puisque nous travaillons en équipe, il ne faut jamais faire un git commit dans la branche principale. Il faut plutôt créer une nouvelle branche (copie de main), ou utiliser une ancienne après l'avoir mis à-jour avec main. Quand vous aurez terminé le développement que vous vouliez faire là-bas, il faut créer un Pull Request avec pour destination la branche main.

Aussi, il faut m’ajouter dans le Pull Request comme l’un des gens qui vont examiner le travail fait. Ceci me permet de m’assurer que la branche principale aura toujours du contenu que je suis okay avec. S’il y a des aspects que je ne suis pas okay avec alors je vais vous demander de faire des corrections. Après que j’aie approuvé le Pull Request vous pouvez la compléter, c’est-a-dire valider l’envoie des modifications faites sur la nouvelle branche dans la branche main.
